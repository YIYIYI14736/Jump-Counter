# 开发日记

## 2026-06-11 YOLO11 跳绳检测稳定性与界面改版

本轮围绕实际手机运行中暴露的问题做了三类收敛：YOLO11 pose 输出稳定、跳绳计数核心算法稳定、前端界面更适合训练场景。

- YOLO11 pose 后处理继续保留按 NCNN 静态输入尺寸解码的方案，避免 v11-n-320/v11-n-640 切换后出现大量错误框和异常骨架线。
- 跳绳计数算法从单纯身体中心点上下阈值，升级为“关键点质量 + 肩/髋融合中心 + 自适应平滑 + 起跳/回落状态机”。当脚踝缺失或 bbox 抖动时，优先用躯干关键点估计人体尺度，降低误计数。
- 针对画面里误检多个人的问题，在 pose 模式下增加主人体锁定。计数和绘制前只保留一个主目标，按关键点质量、与上一帧目标的 IoU/中心距离连续性筛选，减少 v11 模型误检导致的多人干扰。
- 前端采用已确认的 B 方案：相机预览全屏，顶部浮层放标题和摄像头切换，底部浮层放任务/模型/CPU-GPU 选择器，跳绳状态以圆角半透明标签覆盖在画面上。

本轮已完成验证：Android Debug 构建通过；由于本机 MSYS `g++/cc1plus` 异常，独立 C++ host 测试未作为最终验收，Android Gradle/CMake 构建作为本轮主要验证依据。

## 2026-06-09 教师反馈后改为 JumpRope-YOLO

根据新的项目方向，第一版从“桌面防掉落”调整为“基于人体姿态的跳绳计数”。这次调整保留 Android NCNN 示例的摄像头、模型加载和 CPU/GPU/turnip 切换能力，把核心业务逻辑改成：

- Android 默认加载 `pose` 任务，使用 YOLO pose 输出的人体关键点。
- JNI 层新增纯 C++ `JumpRopeCounter`，按身体中心点的低 -> 高 -> 低周期累计跳绳次数。
- Java 层轮询跳绳状态和次数，界面状态条显示 `inactive/no person/ready/counting`，次数增加时短震动并蜂鸣。
- 模型侧目录改为 `JumpRope-Model`，训练包改为 `training.jumprope_yolo`，默认使用 `yolo11n-pose.pt` 和 `datasets/jumprope_pose.yaml`。
- 算法文档改为姿态计数方案，训练文档改为预训练 pose 优先、可选微调和 NCNN 导出。

本轮已完成验证：模型侧单元测试通过，Android Debug APK 构建通过，JNI 计数器纯 C++ 测试通过；当前 Android 代码、JumpRope 模型包和跳绳算法文档中未再保留 DeskGuard 业务逻辑。

## 2026-06-09 第一版任务实现记录

可以，第一版我已经替你往前推进了。先前开的几个旧子代理已经关掉；这轮新开的训练、算法、Android 三个子代理也都处理完了。它们有些只给了部分产物，所以我把剩下的框架、代码、测试和构建自己补齐了。

现在第一版已经具备这些东西：

- Android 端已有“桌面防掉落”本地预警链路：C++ 检测状态、Java 轮询、界面状态条、震动、蜂鸣报警。
- YOLO 微调代码已经写好：支持桌面关键物品检测类，包括 `cup`、`bottle`、`cell_phone`、`laptop`、`keyboard`、`mouse`，也预留了 `desk_edge`、`table_edge` 用于桌面边界识别。
- 边界判断方案已经落成文档：第一版用固定警戒线，后续升级为模型识别桌面边缘。
- “水杯和电脑放得很近”的逻辑也写进方案：把 `cup/bottle` 视为液体风险源，把 `laptop/keyboard/mouse` 视为贵重电子设备，通过中心距离、垂直重叠比例判断 `PROXIMITY_RISK`，用于后续接大模型 TTS 温和提醒。

关键文件在这里：

- Android JNI 状态接口：`yolov8ncnn.cpp (line 118)`
- Java 原生接口：`YOLOv8Ncnn.java (line 22)`
- 主界面报警逻辑：`MainActivity.java (line 240)`
- YOLO 微调入口：`train.py (line 1)`
- 数据集配置：`deskguard.yaml (line 1)`
- 边界与靠近风险算法说明：`deskguard_algorithm.md (line 1)`

目前第一版“怎么判断东西在桌边”的实现是：对杯子、瓶子、手机这类目标取检测框底部中心点，如果这个点连续 5 帧越过画面高度约 `78%` 的警戒线，就判定为 `FALL_RISK`，手机震动并蜂鸣。这个方案适合先跑通 demo，答辩时也容易讲清楚。后续更高级版本再用微调后的 `desk_edge/table_edge` 类真正识别桌沿，把固定警戒线升级为动态桌面边界。

### 日记与 Git 配合规则

1. 每次实现一个阶段性功能后，先运行相关测试或构建命令，把结果写进“验证情况”。
2. 提交 Git 后，用 `git log --oneline -n 5` 查看最新提交号，再把表格里的“待提交”替换成真实提交短哈希。
3. 如果只是修改开发日记、报告或说明文档，可以作为文档提交记录；如果是模型类别、算法、Android 逻辑变化，应单独写清楚功能影响。
4. 回退版本时优先使用 `git revert <提交号>` 生成一个新的反向提交；只想查看旧版本时使用 `git checkout <提交号>`，不要轻易使用 `git reset --hard`。
